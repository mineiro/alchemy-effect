Infrastructure as Effects

Deploy to AWS, Cloudflare, and more with pure TypeScript. Resources, runtimes, and bindings in one type-safe Effect program.

Quick Start ⭐ Star on GitHub
alchemy.run.ts 
import * as AWS from "alchemy-effect/AWS";
import * as Stack from "alchemy-effect/Stack";
import * as Effect from "effect/Effect";

export default Effect.gen(function* () {
  const bucket = yield* AWS.S3.Bucket("DataBucket");
  const putObject = yield* AWS.S3.PutObject.bind(bucket);

  const api = yield* AWS.Lambda.Function("Api", {
    timeout: "30 seconds",
  });

  return {
    bucketArn: bucket.bucketArn,
    apiUrl: api.functionUrl,
    putObject,
  };
}).pipe(
  Stack.make("App"),
  Effect.provide(AWS.providers()),
);

{# ── The Problem ── #}

The Problem

Cloud applications get split across too many systems

Infrastructure definitions, runtime code, permissions, and deployment behavior drift apart until the real architecture is hard to see.

Infrastructure drift

Business logic changes in one place while buckets, queues, functions, and policies change somewhere else.

Hidden wiring

Environment variables, IAM policies, and platform bindings often end up as fragile manual configuration.

Runtime surprises

Eventual consistency, dependency violations, and retries are real cloud behavior, but many IaC tools keep them out of the model.

Lost context

Runtime code is too often configured with strings instead of typed references to the resources it depends on.

{# ── Why alchemy-effect ── #}

Why alchemy-effect?

Infrastructure-as-Effects for modern TypeScript

One program, full stack

Stacks are Effects that declare resources and return outputs. Infra and application composition use the same programming model.

  • Declare resources inline where they are needed
  • Return typed outputs from the same stack definition
  • Keep infrastructure close to business logic
const stack = Effect.gen(function* () {
  const queue = yield* AWS.SQS.Queue("Jobs");
  const worker = yield* AWS.Lambda.Function("Worker");

  return {
    queueUrl: queue.queueUrl,
    workerArn: worker.functionArn,
  };
}).pipe(
  Stack.make("App"),
  Effect.provide(AWS.providers()),
);

Bindings make dependencies explicit

Capabilities connect runtime services with deploy-time permissions so the real dependency graph stays visible and typed.

  • Attach IAM policies and env automatically
  • Bind runtime SDK wrappers as Effect services
  • Model circular dependencies intentionally
const bucket = yield* AWS.S3.Bucket("Assets");
const putObject = yield* AWS.S3.PutObject.bind(bucket);

yield* Http.serve(app).pipe(
  Effect.provide(AWS.S3.PutObjectLive),
);

return {
  main: import.meta.path,
};

Runtime code is part of the resource

Lambda Functions and Cloudflare Workers are first-class resources whose runtime implementation lives in the same model as deployment behavior.

  • Define runtime behavior with Effects
  • Keep config, assets, and code together
  • Support Lambda and Cloudflare runtimes
yield* Http.serve(api);

return {
  main: import.meta.path,
  memory: 1024,
  url: true,
} as const;
}).pipe(
  Lambda.Function("Api"),
);

Effect-native reliability

Errors, retries, and composition are part of the programming model instead of bolted onto deployment after the fact.

  • Bring failures into the type system
  • Use declarative retry logic for cloud behavior
  • Compose infrastructure with ordinary Effect layers
createResource(params).pipe(
  Effect.retry({
    while: isDependencyViolation,
    schedule: Schedule.exponential("1 second"),
  }),
);
